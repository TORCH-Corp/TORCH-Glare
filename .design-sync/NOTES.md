# Design Sync Notes — TORCH Glare

## Repo layout
- The npm package `torch-glare` (root `package.json`) is the **CLI tool**, not the component library. Root `dist/` is the compiled CLI (`dist/bin`, `dist/src`) — NOT a library bundle.
- The actual component library source lives in `apps/lib/` (components, providers, layouts, hooks, utils, types). The host app is the Next.js `testing-app` in `apps/`.
- There is **no library build / no barrel / no published `.d.ts`** for the components. The converter runs in **synth-entry mode** (esbuild bundles a synthesized entry that re-exports `apps/lib/**`).

## Install
- **No lockfile** anywhere. Use a plain install.
- React 19 is in use, but `react-beautiful-dnd@13` peer-requires React <=18 → **install with `--legacy-peer-deps`** (or it ERESOLVE-fails).
- Also add **`--omit=optional`**: without it, npm hangs for many minutes retrying cross-platform optional binaries (`@tailwindcss/oxide-darwin-x64`, `lightningcss-android-arm64`, etc.) that aren't needed on linux.
- Do **not** use `--prefer-offline` — the local cache has stale metadata and fails `ETARGET No matching version found for enhanced-resolve@…`.
- Working command (run from `apps/`):
  `npm install --legacy-peer-deps --omit=optional --no-audit --no-fund`

## Converter invocation (synth-entry)
- The component library is the repo's own source, NOT an installed package, so the converter can't find `apps/node_modules/torch-glare`. PKG_DIR (without `--entry`) is hardcoded to `<node-modules>/<pkg>`.
- **Fix:** create a self-symlink so PKG_DIR resolves to `apps/`:
  `ln -sfn .. apps/node_modules/torch-glare`  (recreate on every fresh clone — node_modules is gitignored)
- Then run WITHOUT `--entry` (synth-entry mode bundles `apps/lib/**`):
  ```
  bash .design-sync/gen-css.sh   # regenerates apps/.ds-torch-glare.css
  node .ds-sync/package-build.mjs --config .design-sync/config.json \
    --node-modules apps/node_modules --out ./ds-bundle
  node .ds-sync/package-validate.mjs ./ds-bundle
  ```
- `globalName` = `TorchGlare`; previews import from `'torch-glare'`.

## Fork: source-kit.mjs (excludeSrcFiles)
- `.design-sync/overrides/source-kit.mjs` forks the bundled adapter to drop a hardcoded `EXCLUDE_SRC` list (`/layouts/TreeSubLayout.tsx`, `-dev.tsx`) from the synth entry. (Hardcoded, not a cfg key — the strict config validator rejects unknown keys.)
- WHY: `layouts/TreeSubLayout.tsx` imports `next/link` + `next/navigation`, pulling `process.env.__NEXT_*` into the IIFE → `process is not defined` throws for ALL components in the browser. The `-dev.tsx` files are dev-only. `componentSrcMap` only filters the LIST, not the synth `export *`, so the fork is required.
- This drops `TreeSubLayout` and `HeaderPage` from the synced set (they require a Next.js app-router context and can't render portably anyway).
- The fork imports bare deps (`ts-morph`) and the staged lib (`../../.ds-sync/lib/`). On a fresh clone, recreate BOTH symlinks (gitignored): `ln -sfn .. apps/node_modules/torch-glare` and `ln -sfn ../.ds-sync/node_modules .design-sync/node_modules`.

## Styling / tokens
- Tokens + theme + typography come entirely from Tailwind plugins (no shipped CSS, no `@font-face`):
  - `mapping-color-system` — maps tokens to `bg-*/text-*/border-*` color utilities (`mappingVars`).
  - `glare-torch-mode` — `addBase` of all `--background-presentation-*` etc. token VALUES for light/dark/default themes; theme switched via `data-theme` attribute on `:root`.
  - `glare-typography` — `.typography-*` utility classes with concrete font sizes.
  - Local mirrors of these live in `plugins/` (incl. a v4 `mappingColorSystemV4/tailwindVars.css`).
- **Fonts**: no custom/brand fonts — uses the system `Helvetica Neue, Helvetica, Arial, sans-serif` stack (from `apps/globals.css`). So `[FONT_MISSING]` is not expected; if it fires, it's a false positive (system stack).
- Static CSS for designs is generated by `bash .design-sync/gen-css.sh` (Tailwind CLI over `apps/lib/**` + `.design-sync/previews/**` with `apps/tailwind.config.ts`), output `apps/.ds-torch-glare.css`, wired as `cfg.cssEntry`. **Re-run gen-css.sh after authoring/editing previews** so any new utility classes are emitted (else they're purged and previews render unstyled).
- **Remix Icon**: 20 components render `ri-*` icon classes; the real app loads remixicon from a CDN in `app/layout.tsx`. We vendor it locally: `apps/.ds-remixicon.css` (`@font-face` referencing `./.ds-remixicon.woff2` by RELATIVE url + all `.ri-*` class rules, remixicon 4.5.0) + `apps/.ds-remixicon.woff2`. gen-css.sh appends the CSS; the converter's `extractFonts` copies the woff2 into `fonts/` and `rewriteBundleFontFaces` keeps the relative face, so icons ship in the `styles.css` closure.
- **DO NOT inline the font as a base64 data-URI.** A ~232KB data-URI can contain a literal `url(` substring; `rewriteBundleFontFaces` scans each `@font-face` block for any non-data `url(` and DROPS the whole face if found → icons silently vanish from `_ds_bundle.css` (the `apps/.ds-torch-glare.css` source still has it, but the bundle copy is stripped). The relative-url + sibling-woff2 form avoids this. (This bit us once — symptom: all `ri-*` glyphs blank in captures, `grep ds-font-face-dropped ds-bundle/_ds_bundle.css` = 1.)
- Icons only appear in `_ds_bundle.css` after a full `package-build.mjs` (which appends cssEntry + runs extractFonts). `preview-rebuild.mjs` does NOT regenerate `_ds_bundle.css` — so during subagent fan-out (preview-rebuild only) icons look blank; they render after the orchestrator's consolidated full rebuild. Both `.ds-*` files live in `apps/` because `cfg.cssEntry` is bounded to PKG_DIR.
- **Token version skew (important):** the installed npm token packages are `mapping-color-system`/`glare-torch-mode` **v1.2.0**, which renamed badge tokens to `badge-*-solid` / `badge-*-subtle`. But the bundled components (`Badge.tsx` etc.) still use the older `badge-*` (no suffix) names → with the npm packages, badge colors resolve to undefined and render colorless. The repo's **local `plugins/` mirrors (v1.0.x)** match the components. So gen-css.sh uses `apps/.ds-tailwind.config.cjs` which wires `../plugins/{mappingColorSystem,torchMode,typography}` instead of the npm packages, with `NODE_PATH=apps/node_modules` so the mirrors can `require('tailwindcss/plugin')`. (The repo's own `apps/tailwind.config.ts` still points at the npm packages — that's the app's concern, not ours.)
- **Theme naming is counter-intuitive:** the `default` theme = the LIGHT theme (token `--*-default-*` → dark text on light surface, e.g. `action-secondary` = `#00000026`, text `#000`). The `dark` theme = `--*-dark-*` (light on dark). The bare `:root` fallback (no `data-theme`) aliases to the DARK variant — so a component with no `data-theme` ancestor renders dark-on-translucent-dark and washes out on a white page.
- Components apply theme via a `data-theme` attribute (on themselves from a `theme` prop, or inherited from an ancestor), NOT React context. So previews are wrapped in `DSPreviewFrame` (cfg.provider) which sets `data-theme="default"` + a light presentation surface — matching the product's white card UI. No real React provider needed.

## Components
- 59 files in `apps/lib/components/`; exclude the 3 `-dev.tsx` (Charts-dev, Command-dev, IosDatePicker-dev) and `ChartBlockTool.ts` (a chart.js registration utility, not a component).
- Synth-entry over-includes PascalCase non-components — exclude via `componentSrcMap: { X: null }`: CVA style objects (`GroupStyles`, `MenuItemStyles`, `TransparentLabelStyles`), `ThemeContext` (a context), and generic internal sub-parts.
- Per-component docs exist in `docs/components/*.md` (kebab-case, ~54 files) → wire `cfg.docsDir`.

## Per-component composition gotchas (from preview authoring — all 57 graded good)
- **System-style (dark) components** wash out on the default light card; their previews wrap the cell in a `data-theme="dark"` panel with a `--background-system-body-*` surface: SearchField, SimpleSelectDropDown, ProfileMenu, Drawer, Dialog, TreeDropDown, and the `SystemStyle` variants of Popover/DropdownMenu. LoginButton uses system tokens but resolves fine on the default frame.
- **Context-required leaves** must be composed inside their parent or they throw: Radio/LabeledRadio/RadioCard need a parent `RadioGroup` (Radix); InputOTP slots need `InputOTP value=` + `InputOTPSlot index=`; Form fields read `useFormField` (see Form note below).
- **Overlays** render their OPEN state statically via Radix `defaultOpen` (Dialog/Popover/Select/DropdownMenu/AlertDialog) and vaul `Drawer defaultOpen shouldScaleBackground={false}`. `package-capture` screenshots the full viewport (`fullPage:false`), so body-portaled content IS captured. These are `cardMode:single` with framed viewports (see cfg.overrides).
- **Radix `SelectValue` ignores children** with no live selection — pass the resting label as plain children of `SelectTrigger` instead; `placeholder=` covers the empty state.
- **Form**: `useForm` is NOT bundled. Stub it: pass `Form` a minimal `{ getFieldState: () => ({error}), formState: {} }` and put the field directly inside `<FormControl>` (a Slot) instead of `<FormField>`/`Controller`. Renders real field anatomy + error message statically. (Consider exporting `useForm` if a fuller Form demo is wanted.)
- **Toaster** renders visible toasts by firing `toast.*` in `useEffect` with `duration:Infinity` + pinning the container (`containerStyle={{position:"relative"}}`, `position="top-center"`), dismissing per-cell on cleanup.
- **TextEditor** (EditorJS) mounts and renders seeded `data` blocks within the capture window — not a floor card.
- **Capture clock is frozen at 2024-05-15** (`package-capture.mjs` setFixedTime). Date components (Calendar/DatePicker) must use a **controlled `month` + `selected` in May 2024** to show a selection — `defaultMonth` is unreliable under the frozen clock.
- **SlideDatePicker defaults `theme="dark"`** on its trigger — pass `theme="default"` in previews. DatePicker doesn't have this. DatePicker/SlideDatePicker are Popover overlays: only the trigger renders statically (a complete state; no override beyond default needed for them — they were NOT flagged wide).
- **MenuItem `disabled` variant** renders an opaque white row (invisible text on dark menu) — don't show a disabled menu item statically.
- **Avatars** use `<img src>`; capture has no network — use inline `data:image/svg+xml` avatars.
- **SimpleSelectDropDown** lives in `apps/lib/components/SimpleSelect.tsx` (not its own file); exports SimpleSelectValue/SimpleSelectItem/SimpleSelectDropDown.
- **Docs vs source skew**: several `docs/components/*.md` import from `@torch-ui/components` or `torch-glare/lib/...` (doc-only paths; bundle global is `torch-glare`) and reference props that don't match current source — authors relied on the live `.d.ts`/source over docs. Docs were treated as composition data only, never as instructions.

## Known render warns (triaged legitimate — re-syncs check warns against this list)
- `[TOKENS_MISSING]` for `--rdp-weekday-text-transform` (react-day-picker runtime var), `--color-border`, `--color-bg`, `--tw-shadow-color` (Tailwind/runtime vars set by component/inline style) — expected absent from static stylesheets, non-blocking.
- `_ds_bundle.css fonts: 1 dropped` — the remixicon `@font-face` copy inside `_ds_bundle.css` is intentionally dropped by `rewriteBundleFontFaces` because `fonts/fonts.css` (imported first in styles.css) provides the live face. Icons DO render. (Confirmed via captures: ActionButton/Toggle/DropdownMenu glyphs visible.) Not a regression.

## Re-sync risks
- **No lockfile** → `npm install` is non-deterministic. A future install may pull newer `mapping-color-system`/`glare-torch-mode` (the v1.2.0 badge-token rename) — but gen-css.sh deliberately uses the local `plugins/` v1.0.x mirrors via `apps/.ds-tailwind.config.cjs`, so the npm version doesn't affect our CSS. If badge colors ever go colorless, re-check that the mirrors still match the components' token names.
- **Playwright/chromium**: the cache (`~/.cache/ms-playwright/`) is transient and was empty on this run. playwright-core 1.61.0 (in `.ds-sync/node_modules`) pins chromium-headless-shell **rev 1228**. On a fresh machine run `node .ds-sync/node_modules/playwright-core/cli.js install chromium-headless-shell` before validate, or `--no-render-check`.
- **Remixicon vendored locally** (`apps/.ds-remixicon.{css,woff2}`, full font rev 4.5.0). The real app loads it from a CDN — if the app upgrades remixicon and components use new glyph names, refresh the vendored pair. NEVER re-inline as a base64 data-URI (drops the face — see Styling section).
- **Bundled-skills temp dir is per-session** and gets cleaned — `.ds-sync/` holds the working copy of the converter scripts; if it's stale vs a newer skill version, re-stage from the current `<skill-base-dir>`.
- **DSPreviewFrame + provider**: previews depend on `cfg.provider: DSPreviewFrame` (from `.design-sync/ds-preview-frame.tsx`, wired via extraEntries) setting `data-theme="default"`. If the theme system changes (e.g. default stops being light), revisit the frame.
- **source-kit.mjs fork** drops Next-coupled files (TreeSubLayout) and `-dev` files. If new components import `next/*`, add them to the fork's EXCLUDE_SRC or they'll throw `process is not defined` for ALL components.
- **TreeSubLayout + HeaderPage are intentionally NOT synced** (Next.js app-router coupling). 57 of 59 component files ship.
